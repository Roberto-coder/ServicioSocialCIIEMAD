# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository layout

The repo root holds only `README.md`, `LICENSE`, an empty `package-lock.json`, and the actual application directory **`paga-web-astro/`**. There is no top-level `package.json` — every npm command must be run from inside `paga-web-astro/`.

```bash
cd paga-web-astro
npm install
```

## Common commands

Run all of these from `paga-web-astro/`:

| Command | Purpose |
| --- | --- |
| `npm run dev` | Start Astro dev server at `http://localhost:4321` |
| `npm run build` | Build to `./dist/` (currently a fully static build — see warning below) |
| `npm run preview` | Preview the built output locally |
| `npm start` | `node dist/server/entry.mjs` — only valid if the project is rebuilt against `@astrojs/node` SSR (it is not, today) |
| `npm run astro -- check` | Type-check `.astro` files |
| `npm run optimize:images` | Dry-run of the image optimizer (Sharp). Reports per-file savings, no writes. |
| `npm run optimize:images:apply` | Re-encode all `public/**/*.{png,jpg,jpeg}` and emit `.webp` siblings. Backs originals up to `public-original/`. |

There is no test runner, linter, or formatter wired up. Don't fabricate one.

## Build / runtime topology — important mismatch

Three pieces of configuration disagree about whether this site is static or SSR. When you touch deploy-adjacent code, pick one model and align all three:

- `astro.config.mjs` sets `output: "static"` and only registers the Tailwind integration.
- `package.json` depends on `@astrojs/node` and exposes `npm start` → `node dist/server/entry.mjs` (an SSR entrypoint that a static build never produces).
- `Dockerfile` runs `npm run build` then `npm start`, so the container will fail to boot as the config stands.

Consequence: API routes under `src/pages/api/` (Open-Meteo proxy, SendGrid contact form) cannot run in production today — a static build either inlines them at build time (`GET` only, once) or skips them entirely. The contact handler is also written using the **legacy lowercase `post` export** (`src/pages/api/contact.js:1`), which Astro 5 no longer recognizes; the modern API requires uppercase `POST`. Note that the live contact form on `/contacto` posts to `https://formsubmit.co/...` (third-party) and ignores `/api/contact` entirely. If you want the in-house endpoint to work, the form action needs to change AND the project needs an SSR adapter AND the export needs renaming.

## Architecture

**Framework.** Astro 5 with Tailwind CSS 3 via `@astrojs/tailwind`. No React/Vue/Svelte components — everything is `.astro` plus inline `<script>` blocks. Swiper is loaded at runtime from a jsdelivr CDN via dynamic `import()` inside page scripts, *not* bundled (see `src/pages/index.astro` and `src/js/swiper.js`).

**Routing.** File-based under `src/pages/`:
- Top-level `.astro` files become pages (`/blog`, `/galeria`, `/podcast`, `/sobre-nosotros`, etc.).
- `src/pages/equipo/[id].astro` is the only real dynamic route — it calls `getStaticPaths()` over `personas` to pre-render one profile per team member.
- `src/pages/posts_blog/[plantilla].astro` is a **deliberate no-op**: it returns `getStaticPaths() => []` so Astro doesn't try to generate dynamic post pages. Real blog posts are hand-authored as `p1.astro` … `p5.astro` in the same folder. Don't add data-driven posts to `[plantilla].astro` — convert the directory to a real collection instead.
- `src/pages/api/` holds API routes (see SSR caveat above).

**Content as plain JS modules.** This project does **not** use Astro Content Collections. Data lives in `src/content/*.js` as exported arrays:
- `personas.js` — team member objects (id, nombre, foto, especialidades, semblanza, enlaces, publicaciones).
- `publicaciones.js` — academic publications (tipo, anio, titulo, autores, enlace).
- `blog.js` — blog post stubs plus `recientes` slice.
- `eventos.js` — calendar entries (one-off `date` or weekly `weekday`).

Author-to-publication linking on the team profile page is fragile: `equipo/[id].astro` filters publications by `pub.autores.includes(persona.publicaciones[0])` — i.e. it matches the **first** entry of `personas.publicaciones` as a substring against the comma-separated authors string. If you add a person, set `publicaciones: ["Lastname-Firstname"]` to a fragment that uniquely appears in their `autores` field.

## Reusable components

Live in `src/components/`. Prefer composing these over inlining the pattern:

| Component | What it replaces / does |
| --- | --- |
| `SmartImage.astro` | Replaces raw `<img>` for **local raster** assets. Emits `<picture><source srcset=".webp" type="image/webp"><img src=".png" ...></picture>` so modern browsers fetch the WebP variant generated by `optimize:images:apply`. Pass `priority` for LCP/above-the-fold images (sets `loading=eager`, `decoding=sync`, `fetchpriority=high`). Skips the `<picture>` wrapper for remote URLs, SVG/GIF/WebP/AVIF, or when `noWebp` is set. Supports `id` passthrough for client-script targeting (used by the mascota crossfade on the home). |
| `IconBadge.astro` | The `bg-{color}/10 p-N rounded-full` icon-circle pattern. Props: `color` (`guinda`/`verde`/`white`), `size` (`sm`/`md`), `pulse`. Icon goes in default slot. |
| `PageHero.astro` | Page header with circular icon + h1 + subtitle. `variant="plain"` (default, light bg) or `variant="gradient"` (verde gradient with white text). Icon goes in `slot="icon"`. Used by `/sobre-nosotros`, `/contacto`, `/nuestro-trabajo`. |
| `FeatureCard.astro` | Three-up homepage card with title + body (`text-justify hyphens-auto`) + CTA. Props: `title`, `color`, `ctaHref`, `ctaLabel`. |
| `SectionTitle.astro` | Centered `<h2>` + optional subtitle for in-page section headers. |
| `SearchOverlay.astro` | Full-site search. See "Search" below. |
| `PublicationCard.astro` | Academic publication card. Maps `tipo` → icon (FileText/BookOpen/GraduationCap) + accent color (verde/guinda). Renders `data-tipo`, `data-anio`, `data-titulo`, `data-autores` attributes consumed by the `/publicaciones` filter script. |

`src/layouts/BlogPostLayout.astro` is the matching layout for `posts_blog/p*.astro` — each post file is just `<BlogPostLayout {...meta}>` plus the prose, no duplicated card/header markup.

## Image strategy

Public-folder images are served as-is by Astro (the `astro:assets` pipeline doesn't touch `public/`). The optimizer at `scripts/optimize-images.mjs` is the substitute:

- It walks `public/` recursively, re-encodes PNG/JPG with Sharp (mozjpeg + palette PNGs), resizes anything wider than `--max-width=1920`, and writes a `.webp` sibling next to each original.
- Originals are copied to `public-original/` on first run with `--apply`.
- `SmartImage` references are what makes the WebP siblings useful at runtime.
- The dry-run on the current asset set reports **61 MB → 14.5 MB re-encoded / 8.3 MB WebP** (76–86% reduction). `banner/ss1.png` was 21 MB; if it's still huge after running the script, re-export it manually.
- If `optimize:images:apply` has not been run, `<source srcset=".webp">` will silently 404 in Network and the browser falls through to the original `<img>` — no visible breakage, but no savings either.

## Layout & global behavior — `BaseLayout.astro`

Wraps every page. Worth knowing:

- **Resource hints** in `<head>`: `preconnect` to jsdelivr (Swiper) and YouTube; `dns-prefetch` to Twitter/Facebook/Instagram; `preload as="image"` for the slider LCP (`/slider/image1i.jpg`). After running `optimize:images:apply` you can switch the preload to the `.webp` for a smaller fetch.
- **Page transition.** Custom fade-in/fade-out: intercepts internal `<a>` clicks, animates `#page-transition`, then `window.location.href`-navigates. Mark a link with `data-no-transition` to opt out (the search results use this so navigation feels instant). External URLs, `mailto:`, `tel:`, `target="_blank"`, downloads, and same-page anchors are skipped automatically.
- **bfcache fix.** A `pageshow` listener calls `showPage()` on every restore — including back-forward cache restores — and `pagehide` resets `#page-transition` to the visible state before the snapshot is taken. Without these, pressing the browser back button left the page blank because `hideAndNavigate` had set `opacity-0` right before the bfcache snapshot was captured.
- **Social SDKs are deferred and conditional.** A small inline script at the end of `<body>` waits for `requestIdleCallback` (or 1.5 s timeout), then injects the Twitter / Facebook / Instagram embed scripts **only if the page contains a corresponding embed selector** (`blockquote.twitter-tweet`, `.fb-page`, `blockquote.instagram-media`, etc.). Don't re-add the SDK `<script>` tags eagerly to `<head>` — that re-introduces the duplicate-load bug that existed previously.
- `loadSocialWidgets()` re-parses embeds on `astro:page-load` and on bfcache restores so client-side navigation keeps them rendering.

## Header & navigation

`src/components/Header.astro` ships:
- Sticky header with responsive logo (`SmartImage`, marked `priority`).
- Desktop menu: `MenuDropdown.astro` (Sobre nosotros) + flat links.
- Mobile menu: hamburger state machine (open/close, animate to X, close on outside-click / Escape / resize ≥ 1024 px).
- Search trigger: rendered in the actions cluster on every breakpoint, opens `SearchOverlay`.

## Search

`src/components/SearchOverlay.astro` is the site-wide search:
- Indexed at build time from three sources: a hand-authored list of static sections, every entry in `personas.js`, and every entry in `blog.js` (`posts`).
- Client-side filtering: tokenized substring match, normalized to strip diacritics, scored (title hits weight 3×, keyword/description hits 1×), top 12 results.
- UI is a full-screen sheet on mobile and a centered panel (`max-w-2xl`) on `sm+`.
- Suggestion chips below the input pre-fill common queries.
- **Empty state CTA:** "¿No encontraste lo que buscabas?" → button to `/contacto`. If you change the contact route, update this component too.
- Keyboard: `/` opens (when not focused in an input), `Esc` closes, click on backdrop closes. Result links carry `data-no-transition` so they bypass the BaseLayout fade for snappier navigation.

## Publications page (`/publicaciones`)

Pre-renders every entry from `publicaciones.js` as `PublicationCard` instances grouped into year sections. All filtering is **client-side, DOM-only** — no rebuild, no fetch:

- Cards expose state via `data-tipo`, `data-anio`, `data-titulo`, `data-autores` (the last two pre-lowercased server-side).
- The script in the page only toggles a `.is-hidden` class on cards and on empty `.year-group` sections, plus updates a `#result-count` and the per-year counters.
- Three combined filters: live text search (debounced 150 ms, normalized to strip diacritics, all whitespace-separated tokens must match), year `<select>`, and a single-select `.tipo-chip` group with a default-active "Todos" chip.
- The filter bar is `sticky top-[4.5rem]` with `backdrop-blur-md`. Year-group headers are also sticky underneath. If you change the header height, both `top-*` values need to move together.
- Empty state element (`#empty-state`) is shown when zero cards match; it offers a "Limpiar filtros" button wired to the same reset routine as the bar's clear-all.
- Keyboard: `/` focuses the search input (skipped when another input/textarea is focused).
- If you add a new `tipo` value to `publicaciones.js`, also add it to the `tipoMap` in `PublicationCard.astro` so it picks up the right icon and accent color (otherwise it falls back to FileText + verde).

## Slider (home)

The Swiper instance on the home page is wired up inside an inline `<script is:inline>` in `src/pages/index.astro`. Notable bits:
- Loaded from CDN via dynamic `import()` of `swiper-bundle.min.mjs` — not bundled with the app.
- A floating pause/play `<button id="slider-toggle">` is rendered `lg:hidden` (mobile/tablet only) and hooks into `swiper.autoplay.start()/stop()` with `aria-pressed`/`aria-label` flipped on each toggle.
- The first slide image uses `SmartImage` with `priority` so it's the LCP candidate and matches the `<link rel="preload">` in BaseLayout.

## Styling system

Tailwind only — `src/styles/global.css` is just the three `@tailwind` directives plus a body reset. The custom palette in `tailwind.config.cjs` is referenced by name across the codebase; don't rename without grepping:
- `guinda` `#6E1942` — primary accent (titles, footer background)
- `verde` `#60A917` — secondary accent (CTAs, hover states)
- Plus `naranja`, `turquesa`, `verdeOscuro`, `verdeProfundo`, `grisAzulado`, `grisOscuro`, `cafe`, `azul`, `mostaza`.
- Fonts: `font-serif` → Playfair Display (titles), `font-sans` → Inter (body). Neither is self-hosted; expect them to come from the browser default unless you add a `<link>` to Google Fonts.

Mobile-first conventions in this codebase: section h1s scale `text-3xl sm:text-4xl md:text-5xl` (don't drop responsive prefixes when adding new heroes — pre-existing pages were corrected for this). Slider/hero heights use `h-[60vh] min-h-[320px] sm:h-[500px]` to avoid eating the viewport on phones.

## Public assets

`paga-web-astro/public/` mirrors deployable URL paths. Notable subfolders: `equipo/` (profile photos referenced by `personas.js`), `galeria/`, `slider/`, `banner/`, `podcast/`, `mascota/`, `logos/`, `redes/`, `events/` (ICS files referenced by `eventos.js`). ## API endpoints

- `GET /api/app` → fan-out to Open-Meteo air-quality + forecast for CDMX (lat 19.43, lon -99.13). Normalizes `hourly` arrays into a synthetic `current` object when the upstream response omits `current`. Consumed by the dashboard cards on `/blog`.
- `POST /api/contact` → SendGrid v3 mail send. Requires `SENDGRID_API_KEY` (mandatory), `CONTACT_FROM_EMAIL`, `CONTACT_TO_EMAIL` env vars. Currently broken (see mismatch section + lowercase `post` export). The live `/contacto` form bypasses this and posts to `formsubmit.co`.

## Deployment

`paga-web-astro/Dockerfile` is a two-stage Node 20 Alpine build: builder runs `npm ci && npm run build`, runner copies `dist/` and runs `npm start`. As noted above, this only works once the project is converted to SSR. For static hosting (Netlify/Vercel/GitHub Pages, the README's stated target), skip the Dockerfile and deploy `dist/` directly.
